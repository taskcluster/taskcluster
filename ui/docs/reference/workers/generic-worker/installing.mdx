# Installing Generic-Worker

The following instructions should be considered more as a guide rather than
concrete requirements. They document _possible_ (and simple) ways to install
and run generic-worker on various platforms. Real world production deployments
may be performed differently.

Generic-worker requires four binaries: `generic-worker` itself,
`start-worker` (worker-runner), `livelog`, and `taskcluster-proxy`.

## Option 1: Obtain prebuilt release

Download the latest releases of `generic-worker`, `taskcluster-proxy`,
`livelog`, and `start-worker` for your platform from
https://github.com/taskcluster/taskcluster/releases.

On darwin/linux, make the binaries executable:

```
chmod a+x generic-worker livelog taskcluster-proxy start-worker
```

## Option 2: Build from source

If you prefer to build from source:

* Download the required Go compiler from https://golang.org/dl/ and follow the
  instructions for your platform. __NOTE: The required Go version is given in
  https://github.com/taskcluster/taskcluster/blob/main/dev-docs/development-process.md#go__.
* Clone the taskcluster monorepo: `git clone https://github.com/taskcluster/taskcluster`
* Build livelog, taskcluster-proxy and start-worker from the `tools/` directory:

  ```
  cd taskcluster/tools
  go install ./...
  ```

  This places the binaries in `$(go env GOPATH)/bin`.

* Build generic-worker from its own directory. It uses a dedicated build script
  because it requires platform-specific build tags and code generation:

  ```
  cd ../workers/generic-worker
  ./build.sh
  ```

  `./build.sh` takes optional arguments: `-a` to build all platforms, and `-t`
  to run tests. By default, only the current platform is built and tests are
  not run. The resulting binary/binaries are placed in the current directory.

## Installation on worker machines

There are two main deployment models:

* __Cloud worker image__: You are building a machine image (AMI, GCP image,
  Azure VM image) that worker-manager will launch on demand. The runner
  configuration is minimal — just the `providerType` and binary paths. At boot,
  worker-runner contacts worker-manager to obtain temporary credentials and
  generic-worker configuration from the worker pool definition.

* __Static or standalone worker__: You are setting up a specific machine
  (datacenter hardware, a Mac mini, etc.) that runs continuously. With the
  `static` provider, the runner configuration includes the worker's identity and a
  shared secret; worker-manager still provides credentials and can provide
  worker config at registration time. With the `standalone` provider (not
  recommended, since it requires permanent credentials stored on disk), all
  credentials and configuration must be provided locally.

Worker-runner assembles the generic-worker configuration file automatically
from multiple sources, in order of increasing precedence: the `workerConfig`
section of the local runner config file, the worker-manager registration
response, and the secrets service. Do not create the generic-worker config
file manually. See the
[worker-runner providers](/docs/reference/workers/worker-runner/providers) and
[worker configuration](/docs/reference/workers/worker-runner/worker-configuration)
documentation for full details.

All platform installations follow the same general pattern:

1. Install the four binaries
2. Generate an ed25519 signing key
3. Create a worker-runner configuration file
4. Configure the system to run worker-runner on boot

__Important__: The generic-worker binary must be readable and executable by the
task user, not only root. If it is not, the worker will exit with exit code 69
(internal error).

### Windows - multiuser build

On Windows, generic-worker is managed by worker-runner (`start-worker.exe`),
with both installed as NSSM services. Worker-runner is set to auto-start and
launches generic-worker (set to demand-start) when work is available.

 1. Install the four binaries (see above):

    | Binary | Location |
    |--------|----------|
    | `generic-worker.exe` | `C:\generic-worker\` |
    | `livelog.exe` | `C:\generic-worker\` |
    | `taskcluster-proxy.exe` | `C:\generic-worker\` |
    | `start-worker.exe` | `C:\worker-runner\` |

 2. Generate an ed25519 private key for signing artifacts:

    ```
    C:\generic-worker\generic-worker.exe new-ed25519-keypair --file C:\generic-worker\ed25519_key
    ```

    The public key will be written to standard out. Keep a copy if you wish to
    validate artifact signatures.

 3. Install [NSSM](https://nssm.cc) (the Non-Sucking Service Manager). The
    easiest way is via [Chocolatey](https://chocolatey.org/):

    ```
    choco install -y nssm
    ```

 4. Install generic-worker as an NSSM service with demand start:

    ```
    nssm install "Generic Worker" C:\generic-worker\generic-worker.exe
    nssm set "Generic Worker" AppParameters ^
        run ^
        --config C:\generic-worker\generic-worker-config.yml ^
        --worker-runner-protocol-pipe \\.\pipe\generic-worker ^
        --with-worker-runner
    nssm set "Generic Worker" Start SERVICE_DEMAND_START
    nssm set "Generic Worker" Type SERVICE_WIN32_OWN_PROCESS
    ```

 5. Install worker-runner as an NSSM service with auto start:

    ```
    nssm install "worker-runner" C:\worker-runner\start-worker.exe
    nssm set "worker-runner" AppParameters C:\worker-runner\runner.yml
    nssm set "worker-runner" Start SERVICE_AUTO_START
    nssm set "worker-runner" Type SERVICE_WIN32_OWN_PROCESS
    ```

 6. Create the worker-runner configuration file `C:\worker-runner\runner.yml`.

    For a __cloud worker image__ (launched by worker-manager), the config is
    minimal — worker-manager provides credentials and generic-worker config
    at boot:

    ```yaml
    provider:
        providerType: azure  # or: aws, google
    worker:
        implementation: generic-worker
        service: "Generic Worker"
        configPath: C:\generic-worker\generic-worker-config.yml
        protocolPipe: \\.\pipe\generic-worker
    cacheOverRestarts: C:\generic-worker\start-worker-cache.json
    ```

    For a __static worker__ (pre-provisioned machine registered with
    worker-manager):

    ```yaml
    provider:
        providerType: static
        rootURL: "https://your-taskcluster-root-url"
        providerID: your-provider-id
        workerPoolID: your-worker-pool/id
        workerGroup: your-worker-group
        workerID: your-worker-id
        staticSecret: "secret-from-worker-pool-definition"
    worker:
        implementation: generic-worker
        service: "Generic Worker"
        configPath: C:\generic-worker\generic-worker-config.yml
        protocolPipe: \\.\pipe\generic-worker
    cacheOverRestarts: C:\generic-worker\start-worker-cache.json
    workerConfig:
        ed25519SigningKeyLocation: "C:\\generic-worker\\ed25519_key"
        livelogExecutable: "C:\\generic-worker\\livelog.exe"
        taskclusterProxyExecutable: "C:\\generic-worker\\taskcluster-proxy.exe"
        taskclusterProxyPort: 8080
    ```

    The `workerConfig` section is where generic-worker settings go. Worker-runner
    assembles the final generic-worker config file automatically — do not create
    it manually. Local `workerConfig` values have the lowest precedence and can
    be overridden by the worker pool definition in worker-manager.

 7. Ensure host 'taskcluster' resolves to localhost by adding the following
    line to `C:\Windows\System32\drivers\etc\hosts`:

    ```
    127.0.0.1 taskcluster
    ```

 8. Reboot the machine. The worker-runner service will start automatically
    and launch generic-worker when work is available.


### macOS - multiuser/insecure build

On macOS, worker-runner is launched by a launchd daemon via a small wrapper
script. The wrapper performs cleanup of stale temporary directories before
starting worker-runner.

 1. Log into the target user account:

    * __Insecure build__: Create a user account `genericworker` and log in as
      that user in a shell.
    * __Multiuser build__: Log in as _root_ (`sudo su -`).

 2. __Multiuser build only__: Disable wizards and warning pop-ups on
    first-login to user accounts:

    * `git clone https://github.com/mozilla-platform-ops/macos-configuration-profiles.git`
    * For macOS versions prior to macOS 11:
      * `profiles install -path=profiles/Skip\ Setup\ Assistant.mobileconfig`
    * For macOS 11 and higher, Apple requires installing the profile via the GUI.

 3. Install the four binaries (see above) to `/usr/local/bin/` and make them
    executable:

    ```
    chmod a+x /usr/local/bin/{generic-worker,taskcluster-proxy,livelog,start-worker}
    ```

 4. Generate an ed25519 private key for signing artifacts:

    ```
    sudo mkdir -p /etc/generic-worker
    ```

    __Insecure build only__: `sudo chown genericworker:staff /etc/generic-worker`

    ```
    /usr/local/bin/generic-worker new-ed25519-keypair --file /etc/generic-worker/ed25519_key
    ```

    The public key will be written to standard out. Keep a copy if you wish to
    validate artifact signatures.

 5. Create the file `/usr/local/bin/run-generic-worker.sh` with the following
    content. The `find` command cleans up temporary directories left behind by
    previous task users that no longer exist on the system:

    ```bash
    #!/usr/bin/env bash

    find /private/var/folders/ -nouser -delete

    cd ~

    /usr/local/bin/start-worker /etc/generic-worker/runner.yml
    ```

    Make it executable:

    ```
    chmod a+x /usr/local/bin/run-generic-worker.sh
    ```

 6. Create `/etc/generic-worker/runner.yml`.

    For a __cloud worker image__, the config is minimal:

    ```yaml
    provider:
        providerType: google  # or: aws, azure
    worker:
        implementation: generic-worker
        path: /usr/local/bin/generic-worker
        configPath: /etc/generic-worker/config
    ```

    For a __static worker__ (such as datacenter hardware), worker-manager
    provides credentials and can also provide generic-worker config from the
    worker pool definition. Local `workerConfig` values serve as defaults
    that the pool definition can override:

    ```yaml
    provider:
        providerType: static
        rootURL: "https://your-taskcluster-root-url"
        providerID: your-provider-id
        workerPoolID: your-worker-pool/id
        workerGroup: your-worker-group
        workerID: your-worker-id
        staticSecret: "secret-from-worker-pool-definition"
        workerLocation:
            host: your-datacenter
    worker:
        implementation: generic-worker
        path: /usr/local/bin/generic-worker
        configPath: /etc/generic-worker/config
    workerConfig:
        cachesDir: "/Library/Caches/generic-worker/caches"
        cleanUpTaskDirs: true
        downloadsDir: "/Library/Caches/generic-worker/downloads"
        ed25519SigningKeyLocation: "/etc/generic-worker/ed25519_key"
        livelogExecutable: "/usr/local/bin/livelog"
        taskclusterProxyExecutable: "/usr/local/bin/taskcluster-proxy"
        taskclusterProxyPort: 8080
        tasksDir: "/Users"
    ```

    The `workerConfig` section is where generic-worker settings go. Worker-runner
    assembles the final generic-worker config file automatically — do not create
    it manually. Local `workerConfig` values have the lowest precedence and can
    be overridden by the worker pool definition in worker-manager.

 7. Create the launch daemon plist at
    `/Library/LaunchDaemons/com.mozilla.genericworker.plist`.

    For a __multiuser__ build (runs as root):

    ```xml
    <?xml version="1.0" encoding="UTF-8"?>
    <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
    <plist version="1.0">
        <dict>
            <key>Label</key>
            <string>com.mozilla.genericworker</string>
            <key>Program</key>
            <string>/usr/local/bin/run-generic-worker.sh</string>
            <key>StandardOutPath</key>
            <string>/var/log/genericworker/stdout.log</string>
            <key>StandardErrorPath</key>
            <string>/var/log/genericworker/stderr.log</string>
            <key>RunAtLoad</key>
            <true/>
            <key>NetworkState</key>
            <true/>
            <key>UserName</key>
            <string>root</string>
        </dict>
    </plist>
    ```

    For an __insecure__ build, change the `UserName` value from `root` to
    `genericworker`.

 8. Install the launch daemon and start it:

    ```
    sudo launchctl load -w /Library/LaunchDaemons/com.mozilla.genericworker.plist
    ```

    Check logs in `/var/log/genericworker/`.


### Linux - multiuser/insecure build

On Linux, worker-runner is run as a systemd service that starts on boot and
manages the generic-worker process.

 1. Make sure your system is up-to-date. For example, on Ubuntu:

    ```
    apt update
    DEBIAN_FRONTEND=noninteractive apt upgrade -yq
    ```

 2. Install curl:

    ```
    apt install -y curl
    ```

 3. For _multiuser_, ensure a desktop environment is installed:

    ```
    apt install -y ubuntu-desktop ubuntu-gnome-desktop
    ```

 4. If tasks require Docker, install it. For example, on Ubuntu:

    ```
    apt install -y apt-transport-https ca-certificates software-properties-common
    curl -fsSL https://download.docker.com/linux/ubuntu/gpg | apt-key add -
    add-apt-repository "deb [arch=amd64] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable"
    apt update
    apt install -y docker-ce
    systemctl enable docker
    ```

 5. Install the four binaries to `/usr/local/bin/`. Replace the version number
    with the desired release:

    ```
    cd /usr/local/bin
    curl -L "https://github.com/taskcluster/taskcluster/releases/download/v99.0.3/generic-worker-multiuser-linux-amd64" > generic-worker
    curl -L "https://github.com/taskcluster/taskcluster/releases/download/v99.0.3/livelog-linux-amd64" > livelog
    curl -L "https://github.com/taskcluster/taskcluster/releases/download/v99.0.3/taskcluster-proxy-linux-amd64" > taskcluster-proxy
    curl -L "https://github.com/taskcluster/taskcluster/releases/download/v99.0.3/start-worker-linux-amd64" > start-worker
    ```

 6. Make binaries executable:

    ```
    chmod a+x /usr/local/bin/{generic-worker,taskcluster-proxy,livelog,start-worker}
    ```

 7. Create directories required by generic-worker:

    ```
    mkdir -p /etc/generic-worker /var/local/generic-worker
    ```

 8. Verify generic-worker runs and shows the expected version:

    ```
    /usr/local/bin/generic-worker --version
    ```

 9. Generate an ed25519 private key for signing artifacts:

    ```
    /usr/local/bin/generic-worker new-ed25519-keypair --file /etc/generic-worker/ed25519_key
    ```

    The public key will be written to standard out. Keep a copy if you wish to
    validate artifact signatures.

10. Ensure host 'taskcluster' resolves to localhost:

    ```
    echo 127.0.1.1 taskcluster >> /etc/hosts
    ```

11. Create the worker-runner configuration file `/etc/start-worker.yml`.

    For a __cloud worker image__ (launched by worker-manager), the config is
    minimal — worker-manager provides credentials and generic-worker config
    at boot:

    ```yaml
    provider:
        providerType: google  # or: aws, azure
    worker:
        implementation: generic-worker
        path: /usr/local/bin/generic-worker
        configPath: /etc/generic-worker/config
    cacheOverRestarts: /etc/start-worker-cache.json
    ```

    For a __static worker__ (pre-provisioned machine registered with
    worker-manager):

    ```yaml
    provider:
        providerType: static
        rootURL: "https://your-taskcluster-root-url"
        providerID: your-provider-id
        workerPoolID: your-worker-pool/id
        workerGroup: your-worker-group
        workerID: your-worker-id
        staticSecret: "secret-from-worker-pool-definition"
    worker:
        implementation: generic-worker
        path: /usr/local/bin/generic-worker
        configPath: /etc/generic-worker/config
    cacheOverRestarts: /etc/start-worker-cache.json
    workerConfig:
        ed25519SigningKeyLocation: "/etc/generic-worker/ed25519_key"
        livelogExecutable: "/usr/local/bin/livelog"
        taskclusterProxyExecutable: "/usr/local/bin/taskcluster-proxy"
        taskclusterProxyPort: 8080
        tasksDir: "/home"
    ```

    The `workerConfig` section is where generic-worker settings go. Worker-runner
    assembles the final generic-worker config file automatically — do not create
    it manually. Local `workerConfig` values have the lowest precedence and can
    be overridden by the worker pool definition in worker-manager.

12. Create a systemd service file at `/lib/systemd/system/worker.service`:

    ```ini
    [Unit]
    Description=Start TC worker
    Wants=network-online.target
    After=network-online.target

    [Service]
    Type=simple
    ExecStart=/usr/local/bin/start-worker /etc/start-worker.yml
    StandardOutput=journal+console
    StandardError=journal+console
    User=root

    [Install]
    RequiredBy=graphical.target
    ```

13. Enable the service so it starts on boot:

    ```
    systemctl enable worker
    ```

14. Reboot the machine, or start the service immediately:

    ```
    systemctl start worker
    ```

    Check logs with:

    ```
    journalctl -u worker -f
    ```
